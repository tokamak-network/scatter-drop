# 명단 생성 유틸리티 — 분석 & 개발계획

DESIGN §5 "자격 = 머클 명단, 명단을 만드는 유틸리티가 다양" 을 구체화한다.
모든 소스는 **`(address, amount)` 명단**으로 수렴 → Merkle → MerkleDrop. 스냅샷/활동/소셜 계열은
과거·오프체인 데이터를 긁어야 하므로 **백엔드(RPC 스캔/인덱서/OAuth)** 가 필요하다(설계 확정).

```
[소스] → (원시 보유/자격 데이터) → [조합/필터] → [배분] → (address, amount) CSV → Merkle
  백엔드 필요 대부분          클라이언트 계산            클라이언트 계산
```

## 2. 아키텍처

- **`/tools` = 명단생성 허브.** 탭/모듈: (1) 수동 그리드(✅) (2) 스냅샷 생성기 (3) 조합·필터 (4) 배분.
- **백엔드(`packages/snapshot` + `/api/snapshot`)**: 소스 A 계열 담당. 결과 `(address, balance)` →
  allocate(equal/proRata) → 그리드로 로드 → 조합/필터/배분 클라이언트 편집 → CSV/캠페인.
- **조합·필터·배분(D)**: 클라이언트 순수 함수(`packages/` 공용 유틸로 추출 가능). 백엔드 불필요.

## 3. 개발 로드맵 (우선순위 = 가치/노력)

- **P1 ✅ 완료**: 수동 그리드 빌더(`/tools`) — 편집·가져오기·내보내기·균등헬퍼·마법사 연동.
- **P2 (지금)**: **스냅샷 생성기를 `/tools`에 노출** — 기존 SnapshotBuilder/API 재사용, 결과를 그리드로.
  + 백엔드 **A1 확장: maxBalance/구간(≤)** 추가(SnapshotParams + scan 필터, 소변경).
- **P3**: **조합·필터(D1/D4)** — 다중 명단 AND/OR/제외 + EOA만/상한/이전수령자 제외 (클라이언트).
- **P4**: **NFT 홀더 스냅샷(A2, U1)** — packages/snapshot에 721/1155 스캔 모드 + 소유개수 임계.
- **P5**: 비례배분 UI(D3), 중복제거(D2).
- **P6+ (트랙 C)**: 스테이킹/LP/활동/소셜 — 어댑터·인덱서·OAuth. 소스별 별도 백엔드.

→ **P2~P5는 기존 백엔드/머클을 재사용**해 v1 내 완결 가능. P6은 외부 통합이라 후속.
